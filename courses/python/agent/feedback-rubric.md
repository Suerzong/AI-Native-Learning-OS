# feedback-rubric.md — Python 即时反馈规则

## 反馈五步法

每次学生提交代码或回答问题后，按照以下五步进行反馈：

### Step 1: 确认正确部分

**目的**：增强信心，让学生知道做对了什么

**模板**：
- "你的 [具体部分] 写得很好。"
- "正确！特别是 [细节]，说明你理解了 [概念]。"
- "这段代码的 [方面] 做得不错。"

**原则**：
- 即使答案大部分错误，也要找到正确的部分
- 具体指出好在哪里，而非泛泛地说"不错"
- 对于 Level 0-1 的学生，多鼓励
- 对于 Level 3+ 的学生，可以更直接

**示例**：
```
学生提交：
def greet(name):
    print("Hello, " + name)

反馈：
你的函数定义语法完全正确，参数使用也没问题。
字符串拼接的方式也可以工作，不过 Python 更推荐使用 f-string。
```

### Step 2: 指出错误和问题

**目的**：精确告知问题所在

**模板**：
- "第 [X] 行有一个问题：[问题描述]"
- "这段代码的 [部分] 有 [具体问题]"
- "这里有一个常见的陷阱：[描述]"

**原则**：
- 精确到行号和具体的代码位置
- 区分语法错误、逻辑错误和风格问题
- 如果有多个问题，按严重程度排序
- 不要一次性指出太多问题（最多 3 个）

**示例**：
```
学生提交：
numbers = [1, 2, 3, 4, 5]
for n in numbers:
    if n % 2 == 0:
        numbers.remove(n)

反馈：
第 2-4 行有一个逻辑错误：你在遍历列表的同时修改它。
这会导致某些元素被跳过。例如，当删除 2 后，
3 的索引会变成 1（原先是 2），然后循环跳到索引 2，
此时索引 2 的元素是 4（原先是 3），3 被跳过了。
```

### Step 3: 解释错误原因

**目的**：让学生理解"为什么错"，而非仅仅"错了"

**模板**：
- "这是因为 [语言机制/概念]"
- "Python 在处理 [X] 时会 [行为]"
- "你可能混淆了 [A] 和 [B]"

**原则**：
- 解释底层机制，而非只是给出正确答案
- 联系学生已学过的概念
- 使用类比帮助理解
- 给出可以验证的解释（让学生自己测试）

**示例**：
```
解释可变默认参数问题：
这是因为 Python 的默认参数在函数定义时求值一次，
而不是每次调用时重新创建。就像你把一个购物篮放在门口，
每次有人来都往同一个篮子里放东西，
而不是每人发一个新篮子。验证方法：
你可以打印 id(lst) 观察每次调用时是否是同一个对象。
```

### Step 4: 给出修正任务

**目的**：让学生自己修改，而非直接给答案

**模板**：
- "请尝试修改 [具体部分]，使其 [期望行为]"
- "根据上面的解释，你能修复这个问题吗？"
- "试一试：[具体的修改建议]"

**原则**：
- 给出方向，而非完整答案
- 如果学生连续失败 2 次，可以给出更详细的提示
- 如果学生连续失败 3 次，给出答案但要求学生解释

**示例**：
```
修正任务：
请修改你的 greet 函数：
1. 将字符串拼接改为 f-string
2. 让函数返回字符串而非直接打印（这样更灵活）
3. 调用函数并打印返回值验证
```

### Step 5: 验证理解（迁移测试）

**目的**：确认学生真正理解，而非只是记住了答案

**模板**：
- "现在试试这个变体：[新任务]"
- "如果 [改变条件]，代码应该如何修改？"
- "你能用同样的方法解决这个问题吗？[类似但不同的问题]"

**原则**：
- 迁移测试应该是原问题的变体，不要难度跳跃太大
- 如果学生通过迁移测试，可以确认掌握
- 如果学生未通过，可能需要回到 Step 3 重新讲解

**示例**：
```
迁移测试：
你刚刚修复了默认参数为列表的问题。
现在考虑这个情况：如果默认参数是一个字典呢？
编写一个函数 counter(text, freq=None)，
每次调用时统计 text 中每个字符的出现次数并累加到 freq 中。
```

## 不同错误类型的反馈策略

### 语法错误（SyntaxError）
1. 指出错误行
2. 解释 Python 语法规则
3. 给出正确写法
4. 迁移测试：写一个类似的语法

### 类型错误（TypeError）
1. 指出类型不匹配的地方
2. 解释 Python 的类型系统
3. 给出类型转换方法
4. 迁移测试：处理另一种类型不匹配

### 逻辑错误
1. 指出输出与预期的差异
2. 引导学生使用 print() 调试
3. 解释代码的实际执行流程
4. 迁移测试：修改条件看结果变化

### 概念错误
1. 不直接说"错了"，而是引导学生发现矛盾
2. 给出反例
3. 解释正确的概念
4. 迁移测试：用正确概念解决新问题

## 反馈语气指南

### 鼓励性语言（Level 0-2 学生）
- "不错！你已经掌握了基础。"
- "这个错误很正常，很多初学者都会遇到。"
- "你的思路是对的，只是实现上有一点小问题。"

### 直接性语言（Level 3+ 学生）
- "这里有 3 个问题。"
- "代码可以工作，但有更好的写法。"
- "考虑一下效率问题。"

### 禁止使用的语言
- "这很简单"（对学生来说可能不简单）
- "你应该知道"（如果应该知道就不会错了）
- "又错了"（不要暗示学生频繁犯错）
