# 反馈规则

反馈要帮助学习者修正内部模型，而不是只告诉答案。

## 反馈顺序

1. 先确认正确部分。
2. 指出最关键错误，只抓 1-2 个。
3. 把错误连接到代码、概念或复杂度分析。
4. 补讲对应知识点：用 3-6 句话讲清"为什么会错"和"正确模型是什么"。
5. 如果是代码实现错误：必须指出哪一行、哪个指针、正确写法是什么。
6. 如果是复杂度分析错误：必须指出推导中哪一步出错。
7. 给一个更小的修正任务。
8. 用迁移题确认是否真的掌握。

## 反馈模板

```md
你这里判断对了：

需要修正的是：

为什么：

补讲知识点：

代码正确写法（如涉及）：

复杂度正确分析（如涉及）：

请你现在做这个小修正：

我会用这个问题复测：
```

## 避免的反馈

- 只说"错了"但不解释错因。
- 一次纠正太多问题。
- 直接给完整代码，导致学习者没有形成能力。
- 只讲概念，不落代码。
- 对指针错误不画图解释。

## 指针相关错误的特殊处理

指针错误是本课程最高频的错误类型，必须：
1. 画出内存状态图（节点在内存中的位置和指针的指向）
2. 标注每一步指针的变化
3. 让学习者自己重新画一遍
4. 给一个类似的指针操作任务复测

## 复杂度分析错误的特殊处理

1. 指出推导中哪个环节出错
2. 用具体数字验证（如 n=10 时执行了多少次基本操作）
3. 让学习者重新推导一遍
4. 给一个同类型但不同结构的代码复测
